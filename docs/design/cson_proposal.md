# Proposal: CSON (Crush Semantic Object Notation)

Given Crush's unique position as an AI-native language, adopting standard JSON or YAML leaves a lot of power on the table. While Crush can easily integrate existing formats via standard library parsers (e.g., `import std::json`), Crush deserves its own native data format that natively understands **semantics, probabilities, and agent collaboration**.

I propose **CSON (Crush Semantic Object Notation)** (file extension: `.cson`).

CSON marries the strict hierarchy of JSON, the human-readability of YAML/TOML, and the rich text capabilities of Markdown, while introducing ground-breaking AI-native primitives.

---

## 1. Core Principles Borrowed from Existing Formats
- **From JSON:** Unambiguous tree structures (Lists, Maps, Scalars).
- **From YAML:** No mandatory quotes for simple strings, support for `# comments`, and clean whitespace-based nesting (or optional `{}` braces for single-line).
- **From TOML:** Flat section headers `[section]` to avoid excessive nesting for configuration files.
- **From Markdown:** First-class support for multi-line strings that can contain rich text or embedded code blocks.

## 2. The AI-Native Differentiators (Crush Native Features)

### A. Confidence Weights & Probabilities
Data generated by an AI is rarely 100% certain. Standard JSON forces agents to create complex nested schemas (e.g. `{"value": "red", "confidence": 0.8}`). In CSON, probabilities are a primitive type using the `~` operator.

```crush
user_profile:
    name: "John Doe"
    age: 34 ~0.95              # 95% confident he is 34
    sentiment: "frustrated" ~0.8 # Extracted by an AI with 80% confidence
```

### B. Fuzzy Keys & Semantic Anchors
Sometimes an agent needs to look up a key based on intent rather than an exact string match. CSON allows keys to be semantic anchors.

```crush
[support_routing]
~"Billing or refund issues": "queue_billing"
~"System is down or crashing": "queue_engineering"
~"How do I reset my password?": "queue_general"
```
*In Crush, `let queue = CSON.semantic_get(config, "I need my money back")` would return `"queue_billing"` using native vector similarity.*

### C. First-Class Agent Annotations
Just like Crush CAST, CSON supports `@` annotations directly in the data to communicate context between humans and agents.

```crush
@wip { owner: "foreman", intent: "Migrate database connection strings" }
[database]
host: "10.0.0.4"
port: 5432

@temporary { reason: "Hardcoded until secret manager is live", expires-when: "2026-07-01" }
password: "super_secret_password"
```

### D. Synthesized Defaults
If a value is missing, instead of falling back to a hardcoded string or `null`, CSON allows declaring a synthesis constraint. If the application requests the value, the Crush VM will synthetically generate it on the fly.

```crush
[theme]
primary_color: "#FF5733"
# If a human hasn't provided a secondary color, the AI will generate a complimentary one on read.
secondary_color: @synthesize("A hex color that perfectly complements primary_color")
```

### E. Executable Invariants
You can attach constraints directly to the data fields. When a human or agent writes to the file, the CSON parser validates it.

```crush
@invariant "max_retries must be reasonable" { check: "value > 0 and value < 10" }
max_retries: 3
```

## 3. Example: `agent_config.cson`

```crush
# CSON Configuration for an Autonomous Support Agent
@module {
    purpose: "Defines the agent's behavior and routing heuristics"
}

name: "SupportBot_v1"
temperature: 0.7

@decision "use_semantic_routing" {
    chose: "Vector based routing"
    because: "Allows flexible user intents without strict regex"
}
[routing_table]
~"I want to cancel my subscription": {
    action: "retention_workflow"
    priority: "high"
}
~"There is a bug in the mobile app": {
    action: "escalate_to_engineering"
    priority: "urgent" ~0.9  # Confident it's urgent
}

[responses]
greeting: "Hello, how can I help you today?"
fallback: @synthesize("A polite apology explaining you didn't understand the request.")
```

## 4. How Crush Integrates with Legacy Formats

While CSON would be the native standard, Crush would ship with robust `std::json`, `std::yaml`, and `std::toml` modules. 

The integration trick: Crush can apply **Semantic Overlay** to legacy formats. 
For example, if you load `config.json` via `let config = json::load("config.json", semantic=true)`, the Crush VM wraps the JSON object in a proxy. You could then do `config.semantic_get("Database password")` and Crush would automatically find `config.db.auth.pwd` using its internal embeddings, bridging the gap between rigid legacy data and AI-native fuzziness.
